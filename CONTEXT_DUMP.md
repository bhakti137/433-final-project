# Context Dump

This document is a full project handoff for another LLM or engineer. It summarizes the current state of the EV Charging Grid Optimization project, what has been built, how it works, where the important files live, what assumptions are in place, and what caveats matter.

## 1. Project Goal

This project builds a locally hosted EV charging optimization dashboard using the dataset:

- `EV_Charging_Grid_Optimization_Categorical.csv`

The main idea is a predict-then-optimize workflow:

1. Load and clean the dataset
2. Engineer basic time and categorical features
3. Train explainable regression models
4. Use those models inside an explicit optimization routine
5. Compare:
   - observed baseline charging
   - dataset-provided optimized charging
   - explicit optimizer output
6. Surface the results in a local Streamlit dashboard

The dashboard is meant to be class-project friendly, easy to explain, and locally runnable.

## 3. Dataset Understanding

### Basic Facts

- Rows: `1000`
- Columns: `21`
- Timestamp range: `2024-01-01 00:00:00` to `2024-01-11 09:45:00`
- Unique timestamps: `1000`
- Unique stations: `19`
- Unique EV IDs: `893`

### Very Important Dataset Caveat

Each timestamp appears once on average.

That means the dataset behaves more like a sequence of independent session snapshots than a rich simultaneous multi-session dispatch problem at each timestamp.

This matters because:

- the optimization is still written as a timestamp-based allocator
- but in practice many timestamps only represent one row/session
- so some optimization-weight sensitivity results may appear very stable simply because there is limited within-timestamp reallocation freedom

This should be remembered when interpreting the optimizer and sensitivity analysis.

## 6. Optimization Logic

### Data Structures

`OptimizationWeights` dataclass now includes:

- `scenario_name`
- `power_loss`
- `voltage_fluctuation`
- `grid_stability`

Default weights:

- `power_loss = 0.45`
- `voltage_fluctuation = 0.35`
- `grid_stability = 0.20`

`BoundsConfig`:

- `min_demand_fraction = 0.80`
- `max_demand_fraction = 1.20`

### Optimization Routine

`optimize_timestamp_allocation()`:

- builds a linear objective using the coefficient of `charging_power` from:
  - `power_loss` model
  - `voltage_fluctuation` model
  - `grid_stability_score` model
- objective:
  - minimize weighted loss effect
  - minimize weighted voltage effect
  - maximize weighted stability effect
- solver:
  - `scipy.optimize.linprog(method="highs")`

### Constraints

- equality constraint on total allocated power matching clipped total predicted demand
- per-row lower bound = `0.8 * predicted_power_demand`
- per-row upper bound = min of:
  - `1.2 * predicted_power_demand`
  - session cap proxy
  - station cap proxy
- station cap estimated from historical 95th percentile load

### Outputs Added by Optimizer

Optimization output dataframe contains:

- `explicit_optimized_power`
- `optimization_feasible`
- `optimization_objective`
- `power_adjustment`
- `recommended_action`

`recommended_action` categories:

- `Increase power`
- `Reduce power`
- `Within target`

Threshold:

- based on `power_adjustment > 0.25`, `< -0.25`, otherwise within target

## 7. Strategy Comparison Logic

The project compares three strategies:

- `charging_power` -> Observed Baseline
- `optimized_charging_power` -> Dataset Optimized
- `explicit_optimized_power` -> Explicit Optimizer

`compare_strategies()` uses model-based evaluation:

- it substitutes the candidate power column into `charging_power`
- then predicts:
  - `eval_power_loss`
  - `eval_voltage_fluctuation`
  - `eval_grid_stability_score`

This is better than the earlier naive scaling approximation that was removed.

## 8. Sensitivity Analysis Implementation

This was one of the latest major additions.

### Purpose

To test whether small changes in optimization weights materially change the optimizer’s recommendations.

### Implemented Scenario Set

Generated by `generate_weight_scenarios()`:

- `default`: `(0.45, 0.35, 0.20)`
- `loss_plus_small`: `(0.50, 0.30, 0.20)`
- `voltage_plus_small`: `(0.40, 0.40, 0.20)`
- `stability_plus_small`: `(0.40, 0.30, 0.30)`
- `loss_minus_small`: `(0.40, 0.40, 0.20)`
- `voltage_minus_small`: `(0.50, 0.25, 0.25)`
- `stability_minus_small`: `(0.50, 0.35, 0.15)`
- `balanced_equal`: `(0.34, 0.33, 0.33)`
- `loss_heavy`: `(0.60, 0.25, 0.15)`
- `voltage_heavy`: `(0.25, 0.60, 0.15)`
- `stability_heavy`: `(0.25, 0.25, 0.50)`

All scenarios are normalized to sum to `1.0`.

### Sensitivity Metrics

Compared against the `default` scenario:

- `action_flip_rate`
- `direction_flip_rate`
- `mean_absolute_power_shift`
- `max_absolute_power_shift`
- `station_rank_change`
- `avg_power_loss_delta`
- `avg_voltage_fluctuation_delta`
- `avg_grid_stability_score_delta`
- `feasibility_rate`

### Stability Rule

A scenario is labeled `Stable` when:

- `action_flip_rate < 0.10`
- `mean_absolute_power_shift < 0.75`
- `feasibility_rate == 1.0`

Otherwise:

- `Sensitive`

Overall optimizer label:

- `Potentially unstable` if 3 or more small-perturbation scenarios are `Sensitive`
- `Reasonably stable` otherwise

Plain-language interpretation:

- stable -> “Small weight changes do not materially alter recommendations.”
- unstable -> “Recommendations are sensitive to minor weight changes and should be interpreted cautiously.”

### Important Interpretation Caveat

Because the dataset has effectively one row per timestamp, many scenarios may produce identical or near-identical outputs.

That means:

- the sensitivity analysis is still implemented correctly
- but a “stable” result can partly reflect the data structure, not necessarily a deeply robust dispatch policy

This is important if another model is asked to improve or critique the method.

### Verification Performed

Reduced-size verification was run successfully:

- default scenario self-comparison produced:
  - `action_flip_rate = 0.0`
  - `mean_absolute_power_shift = 0.0`
- reduced-size sample classified overall optimizer as:
  - `Reasonably stable`

Full 11-scenario runs are heavier and can take noticeable time during app startup.

## 13. Weak Spots / Open Questions

These are the most important areas another LLM might improve next:

1. Model validation is still weak
   - only one random split
   - no time-series validation
   - no grouped/station validation

2. Dataset structure limits realism
   - one row per timestamp on average weakens the dispatch interpretation

3. Optimization objective is coefficient-based and simple
   - it uses only the `charging_power` coefficient from each linear model
   - it does not use a richer full-feature what-if optimization formulation

4. KPI panel still uses simplified scaling logic
   - some dashboard KPI and grid panel calculations still use direct scaling of observed metrics
   - this is less rigorous than the strategy comparison path, which uses model-based evaluation

5. Full sensitivity analysis can be slow
   - the app may benefit from precomputed artifacts or a smaller scenario set for interactive speed

## 14. Suggested Next Best Improvements

If another LLM continues this project, the highest-value next steps are:

1. Add stronger model validation
   - time-based split
   - cross-validation
   - station-group validation

2. Align KPI/grid panel calculations with model-based evaluation
   - remove remaining simple scaling approximations in `render_kpis()` and `render_grid_panel()`

3. Add export/report capability
   - CSV export of recommendations
   - downloadable summary tables

4. Improve optimization realism
   - if better simultaneous session data exists, group true concurrent sessions
   - otherwise document the current limitation even more explicitly

5. Precompute or cache sensitivity artifacts more aggressively
   - serialize outputs to file if startup time becomes a problem
